# data/processed/ contents

Generated by `scripts/04_clean_dedup.py` from `data/raw/faers/` (see
`data/raw/README.md` for the acquisition provenance). Both files here are small
(~40 KB each) and are tracked in git per the project's <5 MB rule for processed data.

## Provenance (2026-07-08 run)

Raw PS-matched report versions: 444
&nbsp;&nbsp;-> deduped to unique cases (FDA-standard: latest FDA_DT, tie-break
highest PRIMARYID): 416
&nbsp;&nbsp;-> removed as present on an FDA quarterly deleted-cases list: 0
&nbsp;&nbsp;-> final analytic cases: 416
&nbsp;&nbsp;-> case-PT rows: 878 (284 unique PTs)

## analysis_cases.csv

One row per unique, deduped, non-deleted case. Columns: `primaryid` (of the
winning/kept version), `caseid`, `quarter` (of the winning version), `sex`, `age`,
`age_cod`, `age_grp`, `wt`, `wt_cod`, `event_dt`, `init_fda_dt`, `fda_dt`, `rept_dt`,
`rept_cod`, `occp_cod`, `reporter_country`, `occr_country`, `outcomes` (`;`-joined
FAERS `outc_cod` values), `serious` (bool, true iff any OUTC row exists for the
case -- FAERS has no separate seriousness flag in DEMO), `n_reactions` (count of
PT rows for the case in `case_pt.csv`).

## case_pt.csv

One row per case-PT pair: `primaryid`, `pt` (MedDRA Preferred Term, verbatim from
REAC), `quarter`, `caseid`, `soc` (System Organ Class -- **currently always null**,
see gap below).

## Known methodological notes (carry into manuscript methods)

1. **Primary Suspect filtering happens before deduplication, not after.**
   `scripts/03_pull_faers.py` (acquisition) already restricts the case universe to
   drug records where datopotamab deruxtecan is `role_cod == "PS"`, before this
   script dedups by caseid. `scripts/04_clean_dedup.py` verifies (via assertion)
   that every case going into dedup is backed by a PS-role record, but it cannot
   independently re-derive a "dedup first, then filter to PS" ordering, because
   non-PS-matching versions of a case were never pulled from FAERS in the first
   place. In practice this makes no difference unless a case's PS/SS role for our
   drug changed between report versions, which was not checked. State this
   ordering explicitly if reviewers ask about it.

2. **MedDRA PT -> SOC mapping is not applied.** MedDRA's hierarchy (PT -> HLT ->
   HLGT -> SOC) is licensed by the MSSO and not freely redistributable, so no
   crosswalk ships with this repo. `case_pt.csv`'s `soc` column is left null. If a
   licensed MedDRA crosswalk becomes available, drop it at
   `refs/meddra_pt_soc.csv` (columns: `pt`, `soc`) and re-run
   `scripts/04_clean_dedup.py` -- it will pick the file up automatically. Until
   then, any SOC-level grouping in the manuscript must be done by hand against a
   licensed MedDRA browser, and the PT-only results should be reported as such.

3. **MedDRA version is not constant across the analysis window.** Confirmed
   27.1 (2025Q1) to 28.1 (2026Q1) via direct inspection of the FAERS XML extract's
   `<reactionmeddraversionpt>` tag (the ASCII files used for the actual pull don't
   carry this tag at all). See `docs/environment.md` for detail.
